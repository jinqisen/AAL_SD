## 0. 当前实现事实（作为“改哪里”的基线）

### 0.1 λ 的实际来源链路

- AD-KUCS 采样中，实际使用的 λ 来自 `ADKUCSSampler._get_adaptive_weight(..., override=lambda_override)`；只要 override 非空，就 **完全等于 override**。  
  代码：[sampler.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/core/sampler.py#L313-L321)
- override 的来源由 `LegacyLambdaProvider.get()` 决定，优先级大致为：  
  `exp_config.lambda_override` → `toolbox.control_state.lambda_override_round`（set_lambda 或 policy 写入）→ `lambda_policy(工具层 apply_round_lambda_policy)` → None。  
  代码：[components.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/experiments/components.py#L100-L130)
- λ-policy 的每轮生成缓存于 `toolbox.control_state["lambda_override_round"]`，并在首次应用时写入 trace：`type="lambda_policy_apply"`。  
  代码：[toolbox.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/agent/toolbox.py#L340-L370)

## 1) λ 轨迹一致性：让“图/表/文/trace”统一口径的落盘改动

目标：让任何一张“λ 轨迹图”都明确表示“采样实际使用的 λ”，并能追溯其来源（override / policy / controller / default）。

### 1.1 在 round_summary 中增加“lambda_effective + lambda_source + lambda_rule”

现状问题：
- trace 里同时可能出现 `lambda_policy_apply`、`lambda_override`、`controller_step.action.lambda`、`round_summary.ranking.lambda_t` 等字段；可视化脚本会选取其中一个（例如 `_effective_lambda` 目前优先 `lambda_override`→`lambda_t`→`lambda_policy_apply`），读者不易理解“哪个才是真正用于采样的 λ”。

建议改动：
- 在每轮 `_append_round_summary` 里写入一个单一权威字段，例如：
  - `lambda_effective`: 本轮采样实际使用的 λ（float）
  - `lambda_source`: {"exp_override","agent_set_lambda","lambda_policy","lambda_controller","sigmoid_default"} 之一
  - `lambda_rule`: 对 policy/controller 的规则名（如 warmup_seeded_uniform / severe_overfit_lambda_down / r2_schedule 等），没有则为 null

落盘位置建议：
- 计算位置：`main.py` 内部在执行 `_query_samples()` 得到 ranked 后，`_last_ranking_metadata` 已存在；这里可将 `lambda_used` 与来源一起塞进 `_last_ranking_metadata`，并最终进入 `round_summary.ranking`。  
  参考：`_last_ranking_metadata` 的使用：[main.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/main.py#L1911-L1950)，round_summary 结构：[main.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/main.py#L781-L823)

验收标准：
- 任何脚本画 λ 轨迹时优先使用 `lambda_effective`，避免歧义。

### 1.2 可视化脚本统一使用 “lambda_effective”

建议改动：
- `analysis/plot_strategy_trajectory.py` 与 `analysis/plot_paper_figures.py`：若存在 `lambda_effective` 字段，直接使用；仅在老 trace 兼容时回退到现有 `_effective_lambda` 逻辑。  
  代码位置：[_effective_lambda](file:///Users/anykong/AD-KUCS/AAL_SD/src/analysis/plot_strategy_trajectory.py#L555-L567)

验收标准：
- 同一 run 的 “策略轨迹图” 与 “round_summary 表/图” λ 曲线完全重合。

### 1.3 在 trace 中显式记录“policy 参数快照”

建议改动：
- 在 run 初始化时追加一次 trace 事件（例如 `type="exp_config_snapshot"`），至少包含：lambda_policy、agent_threshold_overrides、rollback_config、lambda_controller。  
  这样读者可以从单个 run 的 trace 复原关键设置，避免“图里有平台期但不知道阈值/策略是否启用”的争议。

落盘位置建议：
- `main.py` 初始化阶段（创建 toolbox/agent_manager 后、round loop 前），写入一次 `_append_trace({...})`。

## 2) Agent 是否读取图像：两条互斥的落盘路径

### 路径 A（推荐，最小代价）：代码不变，只增强“证据链”

目的：把“Agent 没看图”这件事在 trace 中变成可审计事实，论文也能引用。

建议改动：
- `toolbox.get_system_status()` 的返回 meta 增加字段：`agent_observation_scope = "metrics_only"`。  
  代码位置：[get_system_status](file:///Users/anykong/AD-KUCS/AAL_SD/src/agent/toolbox.py#L485-L566)
- 在 `agent_manager` 每轮写入一条 trace：`type="agent_observation_contract"`，内容包括允许调用的工具列表与是否包含图像内容（false）。  
  代码位置：[agent_manager.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/agent/agent_manager.py)

验收标准：
- 任意 run 的 trace 都能证明 Agent 的输入范围不包含像素/patch。

### 路径 B（如果论文必须宣称“视觉访问”）：新增受控的“图像摘要工具”

注意：这会引入成本、隐私与可复现性问题，需要同时做缓存与日志。

建议新增能力（按从轻到重排序）：
- **B1：通道统计摘要（推荐）**  
  新增工具 `get_sample_channel_stats(sample_id, channels=None)`：返回每通道 mean/std、分位数、极值；不返回像素数组。  
  优点：信息量比全局 mean/std 大；成本可控；不引入图像编码。
- **B2：伪彩合成的低分辨率预览（可选）**  
  新增工具 `get_sample_preview(sample_id, mode="rgb_proxy", size=64)`：从 14 通道选定 3 通道或 PCA->3 维，缩放到小图，base64 PNG 返回。  
  必须同时记录：通道选择策略、归一化方式、尺寸、压缩参数、hash（用于缓存与可复现）。

落盘位置建议：
- 新增/扩展均在 [toolbox.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/agent/toolbox.py) 中实现，并受 `control_permissions` 控制（默认关闭）。
- Prompt 中把工具加入 action space，并显式提示“仅用于解释，不允许把图像内容当作真实标签”。  
  位置：[prompt_template.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/agent/prompt_template.py)

测试建议：
- 在 `src/tests/` 新增或扩展单测，验证返回 JSON schema、数值范围与 deterministic 行为（同一 sample_id 同一结果）。  
  参考：现有 agent/toolbox 单测风格：[test_agent.py](file:///Users/anykong/AD-KUCS/AAL_SD/src/tests/test_agent.py)

## 3) 与本次问题相关的“最小验收清单”

- λ：round_summary 中存在 `lambda_effective`，并且与实际采样 λ 完全一致  
- λ：轨迹图脚本默认使用 `lambda_effective`，不同图/表不再出现口径冲突  
- Agent：明确可审计声明其输入范围（metrics_only 或启用视觉工具的具体形式）
