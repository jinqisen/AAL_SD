# 模型数据加载与评测整改方案（学术严谨 + 工程简洁）

状态：Draft v1（已完成问题梳理与整改设计，待进入实施）

适用范围：AAL_SD 项目（数据加载 / split 口径 / 训练-验证-测试评测 / 主动学习选样链路）

---

## 1. 目标与原则

### 1.1 学术目标（严格）

1. Train/Val/Test 划分口径清晰、不可混用、不可隐式变化，可复现。
2. 评测指标定义严格，不采用“乐观默认值”，避免因边界条件造成指标虚高。
3. 训练过程中对 Val 的使用边界明确：仅用于模型选择/早停/调参时应明确标注为“验证反馈控制”，并保证 Test 只用于最终报告。
4. 禁止“降级方案”掩盖缺失输入或失败路径（例如缺失预测/特征/MC Dropout 时自动回退为 0 分或其他替代计算），以免实验结论被污染。

### 1.2 工程目标（简洁可靠可扩展）

1. 职责拆分：数据集构建、池管理、训练循环、评测、选样的边界清晰，减少 main.py 巨石文件耦合。
2. 配置项“所见即所得”：配置存在即生效，不允许“写了配置但流程完全没用到”。
3. 异常处理严肃：默认 fail-fast；仅在显式配置允许时，才做可控降级，并记录到 manifest/trace 中。
4. 可复现与可审计：每次实验输出可完整重放（seed、RNG、数据指纹、split 口径、关键超参、版本信息）。

---

## 2. 当前实现的关键口径（作为整改基线）

### 2.1 Train/Val/Test 的来源

1. Train/Val/Test 来自数据目录的物理划分（TrainData/ValidData/TestData），并非代码里随机切分得到。
2. 主动学习的 labeled/unlabeled “池”只在 Train split 上生成（初始标注集 + 未标注池），Val/Test 不进入池。

### 2.2 关键风险（整改驱动）

学术口径风险（影响论文/报告可信度）：

1. 轮内指标与实际模型状态可能不一致：如果记录“best epoch 指标”但后续选样使用“last epoch 权重”，会造成“报告更乐观、决策不对应”的口径漂移。
2. 指标实现的边界条件偏乐观：当某类在 y_true 与 y_pred 中均不存在时，IoU/F1 的默认处理可能导致宏平均虚高。
3. 验证集被用作反馈控制信号（overfit risk / rollback / agent 控制），需要明确其研究设定与报告口径，否则会被解读为“对验证集调参过度”。

工程架构风险（影响可维护性与扩展）：

1. main.py 职责过多：池管理、训练循环、选样、agent 控制、恢复/监控等高度耦合，后续引入新 sampler 或新指标会进一步膨胀复杂度。
2. 配置项存在“写入但未生效”的风险（例如 NO_TEST_DURING_TRAINING 目前只出现在 manifest 但不控制流程）。
3. 异常处理存在“吞异常”的点，会导致实验悄悄变成“非预期行为”而不自知（例如某些 try/except: pass）。

## 3. 整改方案总览（分层与模块化）

整改遵循“最小可行拆分”：

1. 先建立严格口径与失败策略（不改算法，先改可靠性）
2. 再抽取模块边界（Pool/DatasetFactory/Evaluator/SelectionAdapter）
3. 最后统一 sampler 协议，减少 pipeline 内 isinstance 分支

### 3.1 建议的目录/模块边界（不追求过度工程）

建议新增或迁移为以下模块（保持简单、文件少、职责单一）：

1. `src/data/dataset_factory.py`
   - 负责构建 train/val/test dataset，验证目录存在性与 mask 约束
   - 统一返回 `DatasetBundle(train, val, test)` 或按需构建

2. `src/data/pools.py`
   - 负责 labeled/unlabeled pool CSV 的创建、加载、保存、回滚
   - 负责 sample_id ↔ index 的映射与完整性校验（union==dataset）
   - 负责数据指纹计算（用于严格恢复）

3. `src/eval/metrics.py`
   - 负责计算 confusion matrix 与指标（mIoU/F1/可选 per-class）
   - 明确“空类”处理策略（ignore 或 NaN-aware mean）

4. `src/train/runner.py`
   - 负责每轮训练：构建 dataloader、训练、评估、保存 checkpoint、记录 trace
   - 输出结构化的 `RoundResult`，供后续 selection/agent 使用

5. `src/selection/adapter.py`
   - 统一 sampler 调用协议，屏蔽 sampler 类型差异
   - 严格保证：缺输入直接抛错，不做静默降级

说明：以上是工程拆分建议；实际实施可先做 `pools.py + metrics.py` 两个文件，把风险最大的口径与指标问题先压住，再拆 runner/adapter。

## 4. 学术严谨性整改（禁止降级 + 明确口径）

### 4.1 严格 split 与数据完整性

1. DatasetFactory 构建 split 时做硬校验：
   - train/val/test 目录必须存在（除非显式允许缺失）
   - train/val 必须有 mask；test 是否有 mask 必须由配置决定（默认“最终评测必须有 mask”）
2. Pools 创建必须绑定到 train split 的“确定样本列表”：
   - 先按文件名排序得到样本全集
   - 再基于 seed 进行稳定抽样生成 labeled/unlabeled
   - 验收：labeled ∩ unlabeled = ∅；labeled ∪ unlabeled = train_all

验收标准：

1. 任意一次 run，pool_integrity_ok 为 true；否则直接失败并写入 status/trace。
2. 任意一次 resume，若数据指纹不一致且 STRICT_RESUME=true，直接失败。

### 4.2 训练口径与“best vs last”一致性

提供两种“严格且可解释”的口径，二选一，必须在 manifest 中固定记录，禁止隐式混用：

1. `MODEL_SELECTION=last_epoch`（工程更简单）
   - 汇报的 round 指标使用 last epoch 在 val 上的指标
   - 后续选样、最终 test 评估使用的也是 last epoch 权重

2. `MODEL_SELECTION=best_val`（学术更常见）
   - 每轮保存 best_val 的 checkpoint
   - 后续选样与最终 test 评估使用 best_val checkpoint 恢复的权重

验收标准：

1. trace 中必须记录每轮使用的 checkpoint（last 或 best_val 的具体路径/标识）。
2. 报告曲线与实际用于 selection 的模型一致（可在 trace 中交叉校验）。

### 4.3 指标计算的严谨性修正

对语义分割的 mIoU/F1，建议采用“空类忽略”策略：

1. 对每个类：
   - 若 `tp + fp + fn == 0`，该类 IoU/F1 记为 NaN 并从宏平均中忽略
2. 报告中同时给出：
   - macro mIoU（忽略空类）
   - per-class IoU（便于审计）
   - 若业务需要，可附带 frequency-weighted IoU

验收标准：

1. 指标实现对“全背景图块”等极端样本不产生虚高的宏平均。
2. 单元测试覆盖：空类、全错、全对、随机混淆四类场景。

### 4.4 禁止降级策略（确保严谨）

统一约束：

1. sampler 必须声明其输入要求（例如 BALD 必须 MC Dropout 预测）。
2. 缺失输入时必须抛出明确异常（例如 MissingMcPredictions、MissingFeatures），不得：
   - 自动回退到 entropy
   - 自动将分数置 0
   - 自动跳过样本并继续产生“看似正常”的排名

验收标准：

1. 在缺失输入时实验状态标记为 failed，并包含清晰 error_type 与 message。
2. 任何“降级路径”若被允许，必须是显式配置打开，并在 trace 中记录 degraded 事件，默认关闭。

## 5. 工程架构整改（简洁可靠可扩展）

### 5.1 main.py 解耦（分阶段）

阶段 A（最小拆分，优先做）：

1. 抽出 PoolManager（pools.py）
   - create/load/save/rollback/integrity/fingerprint
2. 抽出 Metrics（metrics.py）
   - confusion matrix + 严谨 mIoU/F1

阶段 B（清晰职责）：

1. 抽出 TrainerRunner（runner.py）
   - 训练一个 round：训练/验证/产出 RoundResult/保存 checkpoint
2. 抽出 SelectionAdapter（adapter.py）
   - 统一 rank 接口，减少 isinstance 分支

阶段 C（接口统一）：

1. 为所有 sampler 适配统一协议（rank 的输入输出结构一致）
2. pipeline 仅调用 adapter，不感知 sampler 内部差异

### 5.2 配置项治理（防“配置无效”）

整改规则：

1. 配置项必须满足二选一：
   - 被实际逻辑消费
   - 或被移除/标记废弃（并在迁移期给出失败提示）
2. manifest 中记录“配置实际生效值”（effective config），包括自动推导后的值。

重点整改项示例：

1. `NO_TEST_DURING_TRAINING`
   - 若存在该配置：必须在训练流程里实际控制 test 评估是否发生
   - 若 test 只在 final round：该配置应明确为“是否在训练中进行中间 test”，并给出默认策略

### 5.3 复现与审计增强（不复杂化）

1. manifest 记录：
   - seed、deterministic、torch/np 版本、device、split 名称、dataset_fingerprint
   - model_selection 策略（best_val vs last_epoch）
   - 指标版本（metrics schema version）
2. trace 记录关键事件：
   - pool_create / pool_load / pool_integrity
   - round_start / epoch_end / round_end
   - selection（含输入要求、lambda 来源、是否发生降级：默认不允许）

## 6. 异常处理整改（严肃处理，不吞异常）

### 6.1 统一异常策略

默认策略：fail-fast

1. 任何影响实验结论的关键路径失败必须抛出异常并终止当前实验：
   - 数据目录缺失 / mask 缺失（按配置要求）
   - pool 不一致（union != dataset 或 overlap != 0）
   - sampler 输入缺失（features/mc predictions/概率图等）
   - 评测指标非法（NaN 爆炸、维度不匹配）
2. 允许“可选优化失败但不影响结论”的路径可以降级，但必须满足：
   - 显式配置开启，例如 `ALLOW_OPTIMIZATION_FALLBACK=true`
   - trace 中记录 degraded，且最终报告明确标注

### 6.2 异常分级（便于监控与汇总）

建议定义少量异常类型（不要过度细分）：

1. DataError：数据缺失、格式错误、mask 不存在
2. IntegrityError：pool/索引/指纹不一致
3. SamplerInputError：sampler 必需输入缺失或不合法
4. EvaluationError：评测失败或指标异常
5. RuntimeResourceError：OOM、DataLoader worker crash（可重试策略可选，但必须记录重试次数与最终是否成功）

### 6.3 不允许吞异常的点（整改清单）

1. 禁止 `except Exception: pass` 出现在关键路径（训练、评测、选样、池管理）。
2. 若必须捕获（例如写 status/trace 后再抛出），必须：
   - 记录 error_type、message、round/epoch、stack 摘要
   - 重新抛出，避免流程继续执行产生“伪正常结果”

验收标准：

1. 任意关键路径异常都会导致当前实验标记 failed 且退出（除非显式允许重试/降级）。
2. 状态文件与 trace 中能定位到“失败发生在哪一步、为什么、使用了哪些输入与配置”。

## 7. 实施路线图（建议顺序）

### Milestone 1：口径与可靠性优先（强烈建议先做）

1. PoolManager 抽离并强化 integrity/fingerprint 校验
2. 指标 metrics 重写（空类处理严格 + 单测）
3. 明确并实现 model_selection（best_val vs last_epoch）并写入 manifest
4. 清理关键路径吞异常（捕获仅用于补充日志后 re-raise）

### Milestone 2：结构解耦（降低 main.py 复杂度）

1. 拆 DatasetFactory
2. 拆 Runner（round 训练封装）
3. 拆 SelectionAdapter（统一 rank 协议）

### Milestone 3：扩展能力（不牺牲严谨）

1. 统一 sampler 接口与输入要求声明
2. 添加“不可降级”守卫：任何缺输入直接错误，除非显式允许并记录 degraded

---

## 8. 验收清单（最终必须满足）

学术严谨性：

1. train/val/test 口径固定且可追溯（manifest + trace）
2. 指标定义严格且边界条件不虚高（空类处理 + 单测）
3. 报告曲线口径与实际用于 selection 的模型一致（model_selection 固化）
4. 任何 sampler 输入缺失不允许降级（默认 fail-fast）

工程可靠性：

1. main.py 职责显著下降（至少 Pool + Metrics 抽离）
2. 配置项无“名存实亡”，关键配置可通过单测或最小集成测试覆盖
3. 异常不会被吞；失败会终止实验并留下可审计的 status/trace
